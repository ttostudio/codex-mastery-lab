AIDD-Spec v0.1を仕様書にする：AIエージェント開発ワークフローとSaaS化の最初の実演

2026-06-28 / Codex Mastery Lab
対象: AIDD-Spec v0.1 / AI Task Packet / Verification Evidence / AIDD Control Plane MVP
結論: 業界標準の完成版ではないが、標準仕様として議論・検証できるv0.1ドラフトと、AIエージェント実演の最小証跡ができた

先に正直な結論

前回までの成果は、まだ「新しい業界標準の設計ドキュメント」そのものではなかった。

できていたのは、WatchFlow、StudyStream、DrillSwipeを通じて見えた実験手順とナレッジだった。

そこで今回は、記事やAGENTS.mdではなく、仕様書として読める形にまとめ直した。

追加した中心ファイルはこれだ。

standards/aidd-spec-v0.1.md
standards/templates/ai-task-packet-template-v0.1.md
standards/templates/verification-evidence-template-v0.1.md
standards/aidd-control-plane-mvp-v0.1.md
standards/live-build-runbook-001.md

ただし、これで「業界標準が完成した」とは言わない。

正確には、AIDD-Spec v0.1 draft ができた。つまり、これから複数の実験・記事・SaaS化を通じて磨くための、最初の標準仕様である。

AIDD-Spec v0.1とは何か

AIDD-Specは、AIエージェントと人間が同じ説明を読み、同じ完了条件でWeb/モバイルシステムを作るための標準仕様である。

今回のv0.1では、次を定義した。

項目内容
目的AIエージェントに渡す入力、出力、検証証跡、レビュー記録、学習ログを標準化する
適用範囲Web/モバイルアプリ、SaaS MVP、AIエージェント実装、レビュー、検証
成果物モデルProduct / Requirement / Experience / System / Non-Functional / Implementation / Evidence の7層
AI Task PacketAIエージェントへ渡す標準入力
Verification Evidence実行ログ、CI、artifact、スクリーンショットなどの完了証拠
Conformance LevelsL0からL4までの準拠レベル
AIDD Control Planeその流れをSaaS化する方向性

一番大事な考え方

AIDD-Specは「AIに良い感じにお願いするテンプレート」ではない。

AIに渡す前に、少なくとも次を明確にする。

  • 何を作るか
  • 何を作らないか
  • どの状態を持つか
  • どの失敗状態を持つか
  • どのファイルを出すか
  • どのコマンドで検証するか
  • どの証跡を残すか
  • 人間が何をレビューするか
  • 次回どの学びを戻すか

これを1つの最小単位にしたものが、AI Task Packetである。

Conformance Levels

v0.1では準拠度を5段階にした。

Level名前条件
L0Vibe OutputAIが生成しただけ。検証証拠なし
L1Buildable構文/型/buildなど最低限が通る
L2Testableunit/E2Eまたは静的監査で主要条件を検証できる
L3Contract-Verified状態/失敗/API/mockを契約として検証できる
L4CI-EvidencedCI success、artifact、review、learning logが揃う

WatchFlow/StudyStream/DrillSwipeのような公開実験はL4を目標にする。

一方、ライブ配信中に誰でも追える最初の実演では、まずL2を狙うのがよい。重すぎると視聴者が追えないからだ。

今回の実演: IssueBrief Lite

AIDD-Spec v0.1に沿った入力を作り、Codexに渡して、本当に期待したものが出るかを試した。

実験ディレクトリ:

experiments/aidd-spec-v0.1-validation-001/

AI Task Packet:

experiments/aidd-spec-v0.1-validation-001/AI_TASK_PACKET.md

Codexへ渡したPrompt:

experiments/aidd-spec-v0.1-validation-001/CODEX_PROMPT.md

作らせたものは IssueBrief Lite という静的Webアプリである。

目的は、AIに開発タスクを渡す前に、目的・非ゴール・状態・検証条件を1画面で整理すること。

なぜLite実験にしたか

ここでいきなりNext.js + CI + 3ブラウザまでやると、ライブ配信や読者の再現には重い。

まず確認したかったのは、AIDD-Specに沿った入力を渡すことで、AIが次を守れるかである。

  • 指定ファイルを作る
  • 日本語UIにする
  • empty/loading/success/error/offline/timeoutを入れる
  • form labelやaria-liveを入れる
  • 外部通信をしない
  • 静的検査コマンドを用意する
  • review-recordとlearning-logを残す

つまり、今回はL2 Liteの検証である。

Codexに渡した入力の要点

AI Task Packetでは、次を明示した。

入力項目意味
Product BriefIssueBrief Liteが何のためのアプリか
Non-goalsAPI送信、ログイン、DB保存、frameworkを作らない
Scope出力すべきファイルパス
State Contractempty/loading/success/error/offline/timeout
Failure ContractAPI未接続、ログイン不要、課金非ゴール、mediaなし
Accessibility Contractlabel、keyboard、aria-live、ul/li、focus
Quality Gatesnpm run lint:static と npm run test:contract
Verification Evidenceログ、review record、learning log

Codexの出力

Codexは指定通り、次のファイルを作った。

generated-repo/app.js
generated-repo/docs/learning-log.md
generated-repo/docs/product-brief.md
generated-repo/docs/review-record.md
generated-repo/index.html
generated-repo/package.json
generated-repo/scripts/static-contract-check.mjs
generated-repo/styles.css

この時点では、まだ「成功」とは言わない。

AIDD-Specでは、AIの自己申告ではなく、Verification Evidenceを見る。

検証結果

生成後、こちらで独立検証した。

cd experiments/aidd-spec-v0.1-validation-001/generated-repo
npm run lint:static
npm run test:contract

結果:

lint:static passed: 静的ファイルと外部通信禁止ルールを確認しました。
exit=0

test:contract passed: AIDD-Spec L2 Liteの必須契約を確認しました。
exit=0

さらに、独立検査で次を確認した。

independent_leaks_or_external_urls 0
missing_required_tokens []

ブラウザでも開き、入力後にレビュー画面の状態が「送信準備OK」へ変わることを確認した。

今回言えること / 言えないこと

言えること:

  • AIDD-Spec v0.1に沿ったAI Task Packetを渡すと、Codexは期待した出力構造をかなり正確に作れた
  • Non-goalsにより、外部API、ログイン、DB、frameworkへのscope creepを防げた
  • State Contractにより、empty/loading/success/error/offline/timeoutが出力に入った
  • Quality Gatesを指定したため、検証コマンドが生成され、実行できた
  • Verification Evidenceとしてログとreview/learning文書を残せた

言えないこと:

  • AIDD-Specが完成した業界標準である、とはまだ言えない
  • どんなAIエージェントでも同じ品質を出せる、とはまだ言えない
  • L4レベルのCI/artifactまでこのLite実験で確認した、とは言えない
  • SaaS MVPが完成した、とは言えない

今回言えるのは、v0.1 draftとして、仕様→入力→AI実行→検証証跡の最小ループが回った ということだ。

AIDD Control Planeにする

次に作るべきSaaSは、AIDD Control Planeである。

これはAIコーディングエージェントそのものではない。

主価値は、次の流れを誰でも辿れるようにすることにある。

Product Brief
  -> AI Task Packet
  -> Agent Runbook
  -> Verification Evidence
  -> Review Record
  -> Learning Log
  -> Spec Improvement

MVPで作る機能はこうなる。

機能目的
Product Brief Builder何を作るかを短く固定する
AI Task Packet BuilderAIへ渡す入力を作る
Contract Checker必須項目の不足を検出する
Agent Runbook GeneratorCodex等へ渡すコマンドを生成する
Evidence Collector実行ログとartifactを紐づける
Review Record結果を採点する
Learning Log次回改善点を残す

設計ファイル:

standards/aidd-control-plane-mvp-v0.1.md

ライブ配信について

ユーザーの要求は「そのワークフローを誰でも辿れるSaaSを開発していく様子をライブ配信」だった。

ここは制約を明確にしておく。

Hermesだけでは、外部配信プラットフォームで実際に配信開始することはできない。配信先アカウント、OBS、ストリームキー、画面共有権限が必要だからだ。

その代わり、配信でそのまま辿れるRunbookを作った。

standards/live-build-runbook-001.md

Runbookには次を入れた。

  • 配信タイトル案
  • 配信の目的
  • 配信前チェック
  • AIDD-Specを見せる順序
  • AI Task Packetを見せる順序
  • Codex実行コマンド
  • 検証コマンド
  • SaaS MVPとして何を作るか
  • 次回に作る画面

配信で見せる流れ

配信では、以下の順に見せる。

1. AIDD-Spec v0.1を見る
2. AI Task Packetを見る
3. Codexへ渡す
4. 出力されたIssueBrief Liteを見る
5. npm run lint:static / test:contractを実行する
6. Verification Evidenceを見る
7. AIDD Control Plane MVPの設計を見る
8. 次回、SaaSの最初の画面を作る

ここまでで、AIDD-Specの考え方を「記事上の概念」ではなく「実行できる流れ」として見せられる。

次にやるべきこと

次回は、AIDD Control Plane MVP自体をNext.jsで作り始める。

最初のAI Task Packetは次の範囲に絞る。

/projects/new
/packets/new
/packets/:id/runbook
/packets/:id/review

最初の合格条件は次でよい。

  • Product Briefを入力できる
  • AI Task Packet YAMLを生成できる
  • 必須項目不足を表示できる
  • Codexへ渡すRunbookを生成できる
  • Review RecordとLearning Logの雛形を作れる

これはまだSaaS完成ではない。

しかし、AIDD-Specに沿った開発ワークフローを、SaaSの画面に落とし込む最初の一歩になる。
